Fastjson 1.2.24

在2017年3月15日，fastjson官方主动爆出在 1.2.24 及之前版本存在远程代码执行高危安全漏洞。

影响版本：fastjson <= 1.2.24 描述：fastjson 默认使用 @type 指定反序列化任意类，攻击者可以通过在 Java 常见环境中寻找能够构造恶意类的方法，通过反序列化的过程中调用的 getter/setter 方法，以及目标成员变量的注入来达到传参的目的，最终形成恶意调用链。

此漏洞开启了 fastjson 反序列化漏洞的大门，为安全研究人员提供了新的思路。我也有点落后的来学习了😶‍🌫️

TemplatesImpl 利用链

之前cc3的时候有利用过TemplatesImpl加载字节码，这里其实也是一个道理。

TemplatesImpl类实现了Serializable接口，因此可以被序列化。

首先我们关注该类中的一个成员属性_class，是一个 Class 类型的数组，数组里下标为 _transletIndex 的类会在 getTransletInstance() 方法中使用 newInstance() 进行实例化。

哪里可以调用这个getTransletInstance()呢？

我们找到这样一条调用链 getOutputProperties() => newTransformer() => getTransletInstance()

而getOutputProperties()方法就是类成员变量_outputProperties的getter方法，我们可以在fastjson处理@type的json时调用。这就给了我们一条很容易触发的利用链。

那么这个_class可控吗？

搜索一下调用，发现在readObject、构造方法以及defineTransletClasses()中有赋值操作

defineTransletClasses() 在 getTransletInstance() 中有调用，如果_class 不为空则会被调用

跟进一下看一下defineTransletClasses()

要求 _bytecodes 不为null，然后调用自定义的ClassLoader去加载 _bytecodes 中的 byte[]；_bytecodes 也是该类的成员变量

如果这个类的父类为 ABSTRACT_TRANSLET 也就是 com.sun.org.apache.xalan.internal.xsltc.runtime.AbstractTranslet，就会将_transletIndex设置为i，如果是第一次调用，那么_transletIndex = 0 得到_class[0]

不过父类不是 ABSTRACT_TRANSLET，则在后面的判断中抛出异常（默认情况下_transletIndex = -1）

完整利用链

这时候一条完整的利用链就出来了：

我们需要通过@type让Fastjson反序列化一个TemplatesImpl 类，其中_bytecodes 是我们构造的恶意类的字节码，这个类的父类是 ABSTRACT_TRANSLET，最终这个类会被加载并使用 newInstance() 实例化。

在反序列化过程中，getter方法getOutputProperties()会被fastjson调用，这个方法将触发整个利用链：getOutputProperties() => newTransformer() => getTransletInstance() => defineTransletClasses() => EvilClass.newInstance()

其中，为了满足漏洞点触发之前不报异常及退出，我们还需要满足 _name 不为 null ，_tfactory 不为 null

由于我们需要更改的私有变量（_bytecodes _class）没有setter方法，在反序列化时需要使用 Feature.SupportNonPublicField 参数。这让这条利用链变得非常鸡肋。
